# Push Notifications - Design Document

**Ticket:** KAI-105
**Status:** Design
**Author:** Commercial Platform Team
**Date:** 2026-04-03

---

## 1. Overview

Push Notifications deliver real-time alerts from MediaMTX NVR servers to users' mobile devices (iOS and Android) and web browsers. Notifications cover events such as camera offline, storage warnings, motion alerts (future), and system health issues.

## 2. Goals

- Deliver time-sensitive NVR alerts to mobile devices via FCM (Android) and APNs (iOS)
- Support web push notifications via FCM for browser-based users
- Allow per-user notification preferences and quiet hours
- Implement rate limiting to prevent notification fatigue
- Provide reliable delivery with retry and fallback mechanisms

## 3. FCM / APNs Integration

### 3.1 Architecture

```
NVR Server --> Cloud Notification Service --> FCM --> Android / Web
                                          --> APNs --> iOS
```

- NVR servers do not communicate directly with FCM/APNs. They publish events to the Cloud Management Portal (KAI-103), which routes notifications through a dedicated Notification Service.
- For standalone (non-cloud) deployments, the NVR server includes an embedded notification sender that communicates with FCM/APNs directly using configured credentials.

### 3.2 FCM Integration (Android + Web)

| Item | Detail |
|------|--------|
| Protocol | FCM HTTP v1 API (`https://fcm.googleapis.com/v1/projects/*/messages:send`) |
| Auth | Google Service Account (OAuth 2.0 bearer token) |
| Credential storage | Service account JSON stored encrypted in cloud config; for standalone, in NVR's SQLite `settings` table (AES-256 encrypted with `nvrJWTSecret`) |
| Token management | Client app registers FCM token on login; tokens stored per-user-device |

### 3.3 APNs Integration (iOS)

| Item | Detail |
|------|--------|
| Protocol | APNs HTTP/2 (`api.push.apple.com:443`) |
| Auth | Token-based (JWT with APNs key) -- preferred over certificate-based |
| Key storage | APNs key (.p8) stored encrypted alongside FCM credentials |
| Token management | Client app registers device token on login; stored per-user-device |

### 3.4 Device Token Lifecycle

```
Device Registration:
  Client App --> POST /v1/notifications/devices
  {
    "platform": "ios" | "android" | "web",
    "token": "<fcm_or_apns_token>",
    "device_name": "iPhone 15 Pro"
  }

Token Refresh:
  Client App --> PUT /v1/notifications/devices/:id
  { "token": "<new_token>" }

Unregister:
  Client App --> DELETE /v1/notifications/devices/:id
```

- Stale tokens (APNs feedback / FCM `UNREGISTERED` error) are automatically purged.

## 4. Notification Payload Format

### 4.1 Internal Event Schema

Events generated by the NVR:

```json
{
  "event_id": "uuid",
  "server_id": "uuid",
  "type": "camera.offline",
  "severity": "warning",
  "timestamp": "2026-04-03T14:30:00Z",
  "camera_id": "lobby-entrance",
  "camera_name": "Lobby Entrance",
  "site_label": "headquarters",
  "details": {
    "last_seen": "2026-04-03T14:28:30Z",
    "error": "RTSP connection timeout"
  }
}
```

### 4.2 Event Types

| Type | Severity | Description |
|------|----------|-------------|
| `camera.offline` | warning | Camera stopped responding |
| `camera.online` | info | Camera reconnected |
| `camera.recording_failed` | critical | Recording could not be saved |
| `storage.warning` | warning | Disk usage exceeds threshold (default 85%) |
| `storage.critical` | critical | Disk usage exceeds critical threshold (default 95%) |
| `server.offline` | critical | NVR server unreachable (cloud-only) |
| `server.restart` | info | NVR server restarted |
| `system.update_available` | info | New version available |
| `alert.custom` | varies | User-defined alert rules (future) |

### 4.3 FCM Payload

```json
{
  "message": {
    "token": "<device_token>",
    "notification": {
      "title": "Camera Offline: Lobby Entrance",
      "body": "Lobby Entrance at Headquarters has been unreachable since 2:28 PM."
    },
    "data": {
      "event_id": "uuid",
      "event_type": "camera.offline",
      "server_id": "uuid",
      "camera_id": "lobby-entrance",
      "deep_link": "mediamtx://cameras/lobby-entrance"
    },
    "android": {
      "priority": "high",
      "notification": {
        "channel_id": "nvr_alerts",
        "icon": "ic_notification"
      }
    },
    "webpush": {
      "headers": { "Urgency": "high" },
      "notification": {
        "icon": "/icons/alert-192.png",
        "badge": "/icons/badge-72.png"

```

## 5. User Preferences

### 5.1 Preference Model

```
NotificationPreferences
  |-- user_id (FK)
  |-- enabled (bool, default true)
  |-- event_types: {
  |     "camera.offline": true,
  |     "camera.online": false,
  |     "storage.warning": true,
  |     "storage.critical": true,
  |     "server.offline": true,
  |     ...
  |   }
  |-- severity_filter: "warning" | "critical" | "all"
  |-- quiet_hours: {
  |     "enabled": true,
  |     "start": "22:00",
  |     "end": "07:00",
  |     "timezone": "America/New_York",
  |     "override_critical": true  // critical alerts bypass quiet hours
  |   }
  |-- per_camera_overrides: [
  |     { "camera_id": "parking-lot", "muted": true }
  |   ]
```

### 5.2 Preference API

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/v1/notifications/preferences` | Get current user's preferences |
| PUT | `/v1/notifications/preferences` | Update preferences |
| POST | `/v1/notifications/preferences/test` | Send a test notification |

### 5.3 Preference Evaluation Order

1. Check if notifications are globally enabled for the user.
2. Check if the event type is enabled.
3. Check severity filter.
4. Check per-camera mute overrides.
5. Check quiet hours (critical events bypass if `override_critical` is true).
6. If all checks pass, send notification.

## 6. Rate Limiting

### 6.1 Rate Limits

| Scope | Limit | Window | Behavior |
|-------|-------|--------|----------|
| Per user | 30 notifications | 1 hour | Excess notifications are batched into a digest |
| Per user per event type | 5 notifications | 15 minutes | Subsequent events are suppressed with a count |
| Per camera | 3 notifications | 10 minutes | Flapping cameras do not spam the user |
| Global per server | 100 notifications | 1 hour | Prevents runaway event storms |

### 6.2 Rate Limit Behavior

- When a rate limit is hit, notifications are **batched** rather than dropped.
- A digest notification is sent at the end of the window: "5 more alerts from Headquarters in the last 15 minutes."
- Critical severity events (`server.offline`, `storage.critical`) are exempt from per-event-type and per-camera limits but still subject to the per-user hourly limit.

### 6.3 Implementation

- Rate counters are stored in Redis (cloud) or in-memory (standalone) with TTL-based expiration.
- Each notification attempt increments the relevant counters and checks limits before dispatching.

## 7. Delivery Reliability

- **Retry policy:** Failed FCM/APNs sends are retried 3 times with exponential backoff (1s, 5s, 25s).
- **Dead letter queue:** After 3 failures, the notification is logged to a DLQ table for manual review.
- **Delivery tracking:** Each notification is logged with delivery status (`sent`, `delivered`, `failed`, `rate_limited`).
- **Token cleanup:** Tokens that return `UNREGISTERED` or `BadDeviceToken` are removed from the device registry.

## 8. Standalone Mode

For NVR servers not connected to the cloud portal:

- The NVR server runs the notification sender directly.
- Admin configures FCM/APNs credentials in the local admin console.
- Device tokens are stored in the local SQLite database.
- Rate limiting uses in-memory counters.
- All other features (preferences, payloads, event types) work identically.
